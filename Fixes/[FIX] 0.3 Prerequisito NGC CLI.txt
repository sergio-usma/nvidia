Alright, let's step back and look at the logs. There are two issues happening here:

1. **Why it says `-bash: /usr/local/bin/ngc: No such file or directory**`:
When you ran `sudo mv ngc-cli/ngc /usr/local/bin/` in your very first step, you stripped the `ngc` binary out of its original directory. When you later moved `~/Downloads/ngc-cli` to `/usr/local/share/ngc-cli`, it was moved **without** the `ngc` binary inside it. Your symlink points to a file that isn't there anymore.
2. **PyInstaller's Behavior**:
PyInstaller binaries often rely on `sys.argv[0]` or specific environment layouts. If it is symlinked across different system root directories, it can sometimes get confused about its original path.

The cleanest and most foolproof way to fix this is to delete the broken pieces, re-extract the folder so it is completely intact, and simply add it to your user's `PATH` variable instead of trying to move files around.

Run this sequence of commands to reset and fix it:

### 1. Clean up the broken files

```bash
sudo rm -f /usr/local/bin/ngc
sudo rm -rf /usr/local/share/ngc-cli

```

### 2. Re-extract the original download to get a clean, complete folder

```bash
cd ~/Downloads
unzip -o ngccli_arm64.zip

```

### 3. Move the intact directory to `/opt`

Keeping the entire directory together in `/opt` is standard practice for manual software installations on Linux:

```bash
sudo mv ngc-cli /opt/ngc-cli
sudo chmod +x /opt/ngc-cli/ngc

```

### 4. Create an alias or add it to your PATH

Instead of a symlink (which tricks PyInstaller's path detection), we will add it directly to your shell profile.

Run this command to append it to your `.bashrc`:

```bash
echo 'export PATH="/opt/ngc-cli:$PATH"' >> ~/.bashrc

```

### 5. Refresh your terminal and test

```bash
source ~/.bashrc
```

Don't worry, your ngc installation is actually working perfectly now! What you are seeing here isn't a failure of the NGC tool, but a standard Linux behavior when using head.

Why this happens
When you pipe output into head -5, head only wants to read the first 5 lines. As soon as it gets them, it immediately closes the pipe and exits.

Because ngc registry model list prints a massive JSON list, it is still trying to write data to the pipe after head has already shut it down. Python catches this sudden closure and spits out [Errno 32] Broken pipe.

The Solution
If you just want to inspect the data or see a snippet of the models without getting the broken pipe error, you can pipe it into less instead, which handles the stream gracefully:

Bash
ngc registry model list | less
(Press q to exit less when you're done browsing).
